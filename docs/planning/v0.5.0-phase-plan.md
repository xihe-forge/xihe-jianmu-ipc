## Phase 1：IPC 同名接力（release-rebind）

STATUS: implemented 2026-04-19

**范围**：Hub 端独立闭环，不依赖其他 Phase。

### 实现

**db schema**（lib/db.mjs）：
```sql
CREATE TABLE IF NOT EXISTS pending_rebind (
  name TEXT PRIMARY KEY,
  last_topics TEXT,         -- JSON array
  buffered_messages TEXT,   -- JSON array 宽限期内到来的消息
  released_at INTEGER NOT NULL,
  ttl_seconds INTEGER NOT NULL DEFAULT 5,
  next_session_hint TEXT    -- 继任者提示（可选）
);
CREATE INDEX idx_pending_rebind_released_at ON pending_rebind(released_at);
```

**端点**（lib/http-handlers.mjs）：

- `POST /prepare-rebind`
  - body: `{name: string (必填), ttl_seconds?: number (默认 5), topics?: string[] (主动列出要继承的 topic，缺省用当前 session 的订阅), next_session_hint?: string}`
  - 行为：将 session 标记为 "clean handover pending"（内存 Map + DB 预写），返回 `{ok, will_release_at}`
  - 错误：name 不是当前 session 的 → 403；已有 pending_rebind 未过期 → 409

**WebSocket 断连处理**（lib/router.mjs close handler）：
- 检查是否有 prepare-rebind 记录
- 有 → 写 `pending_rebind` 表，`released_at = now`
- 无 → 走原有离线逻辑（inbox flush 到 SQLite、session 移除）

**宽限期内消息处理**：
- 消息路由到该 name 时，Hub 先查 `pending_rebind`，有且未过期 → 追加到 `buffered_messages` JSON 数组
- 超过 `ttl_seconds` → 清理记录，消息走正常 stub / inbox 逻辑

**新 session 连入**（WebSocket auth `name=X`）：
- 查 `pending_rebind` where name=X
- 命中且未过期 → 恢复 topic 订阅 + flush `buffered_messages` + 删除 pending_rebind 记录
- 未命中 / 过期 → 新 session 占用 name 无继承（不破坏现有行为）

**清理 job**：setInterval 每 5s 扫 `pending_rebind` 删除 `released_at + ttl_seconds * 1000 < now` 的记录（复用 Phase 3 scheduleInboxCleanup 模式）。

### 测试用例（6 场景，tech-worker 14:09 列）

1. 正常 rebind：A 发 prepare-rebind → 断连 → B 以同名连入（2s 内）→ 继承 topic + 收到缓冲消息
2. 5s 超时清缓冲：A prepare-rebind → 断连 → 无人连入 5s+ → 记录清理 + 缓冲消息 drop
3. 同名双连竞争：A 未 prepare-rebind 就断 + 立刻 B 同名连入 → B 占用但无继承
4. inbox 流到新 session：A prepare-rebind 前有未读 SQLite inbox → B rebind 后 inbox 与 buffered 一起 flush（顺序：旧 inbox 先，buffered 后，按 ts 升序）
5. topic 订阅继承：A 订阅了 `network-up` + `custom-topic` → B rebind 后两个 topic 都继承
6. 密钥校验：WebSocket auth 层保持现有 token 校验不破（新 session 必须有效 token 才能用 name）

### Commit 切分（4 commits）

## Phase 2：ipc_spawn 扩 host 参数

STATUS: implemented 2026-04-19

**范围**：MCP tool + HTTP /api/spawn 双入口扩字段。独立小改，解锁 Phase 4。

### 字段设计

**MCP tool schema**（lib/mcp-tools.mjs）：
```typescript
{
  name: "ipc_spawn",
  inputSchema: {
    type: "object",
    properties: {
      name: { type: "string", description: "新 session 名，必填" },
      task: { type: "string", description: "初始任务，必填" },
      interactive: { type: "boolean", default: false },
      model: { type: "string", enum: ["opus", "sonnet", "haiku"], default: "sonnet" },
      host: {
        type: "string",
        enum: ["wt", "vscode-terminal", "external"],
        default: "external",
        description: "spawn 载体：wt=Windows Terminal 新 tab / vscode-terminal=VSCode 终端 / external=不自动 spawn，调用方自行"
      }
    },
    required: ["name", "task"]
  }
}
```

**HTTP 端点** `POST /api/spawn` body 同上，多一个 `host` 字段，默认 `external` 兼容现有调用。

### 实现分发

- `host === "wt"`：调 `wt.exe new-tab --title <name> --starting-directory <cwd> cmd /c "set IPC_NAME=<name> && \"<claude-bin>\" --dangerously-skip-permissions --channels server:ipc"`；session 名通过 env 传入，不使用 `--session-name` / `--resume`
- `host === "vscode-terminal"`：通过 VSCode tasks.json 机制触发（tech-worker 提供 tasks 模板 + 触发方式；本 Phase 可能暂留 TODO，文档写清 tech-worker 模板落地后再接，或降级为 external）
- `host === "external"`：不 spawn，返 `{spawned: false, host: "external", hint: "caller should spawn session externally"}`，保持现状

### 兼容性

- 旧调用不传 host → 默认 `external`，**行为完全不变**
- response schema 加 `host` 字段（返回实际生效的 host）
- tech-worker 的 `scripts/spawn-successor.sh` `build_spawn_request_body` (line 454-465) 加 host 字段，mock 的 `assert_spawn_requests` 同步断言

### Commit 切分（2 commits）
1. `feat(api): ipc_spawn 扩 host 参数（wt/vscode-terminal/external），HTTP + MCP 双入口`
2. `docs(CLAUDE.md): ipc_spawn 工具说明更新 + spawn-successor.sh 联调指引`

## Phase 3：watchdog harness-heartbeat probe

STATUS: implemented 2026-04-19

**范围**：network-watchdog 加第 5 个 probe。独立于 Phase 4 实现（只检测，不触发 handover）。

### Probe 设计

**新文件** `lib/harness-heartbeat.mjs`：
```js
export async function probeHarnessHeartbeat({
  hubUrl = 'http://127.0.0.1:3179',
  timeoutMs = 5000,
  maxSilentMs = 600000  // 10 分钟无响应算异常
} = {}) {
  // 1. 查 /sessions 看 harness 是否在线
  //    - 不在线 → ok: true（离线不算 watchdog 问题，harness-daemon 管）
  // 2. 在线 → 查 /messages?peer=harness&limit=1 拿最近一条消息 ts
  //    - now - lastMsgTs > maxSilentMs → ok: false, error: "silent"
  //    - 否则 → 发 ping 消息，期 5s 内 ack（复用 router ackPending 机制）
  //      - ack 到 → ok: true
  //      - timeout → ok: false, error: "no-pong"
  // 返回 {ok, latencyMs, error?, detail: {lastMsgAge, pingAckMs, contextWarnLevel}}
}
```

**额外监听**：harness 主动 IPC 消息 `[context-warn] <pct>%`
- 实现方式：watchdog 订阅 `harness-context` topic
- harness 自报 55%+ → watchdog 记录 contextWarnLevel
- harness 自报 65%+ → probe 直接返 `ok: false`（触发 self-handover 由 Phase 4 接）

### 状态机接入

- 现有 network-state.mjs `probes` 扩 5 项
- **但 harness 挂 ≠ 网络挂**：广播不走 `network-down` topic（那是外部依赖用的），新增 `harness-down` topic
- 扩展 `createStateMachine` 或新建 `createHarnessStateMachine`，或直接在 network-state 内区分 "component" 维度

**推荐**：新建 `lib/harness-state.mjs`，独立状态机，复用探测+阈值思路但**不和外部依赖混**。好处：两者失败语义不同，合并会让代码过于复杂。

### 测试

- 单元测试 `tests/harness-heartbeat.test.mjs`：
  - harness 离线 → ok: true（忽略）
  - harness 在线 + 最近有消息 + ping 通 → ok: true
  - harness 在线 + 最近消息 > 10min → ok: false, silent
  - harness 在线 + ping 超时 → ok: false, no-pong
  - context-warn 65% 收到 → ok: false, context-overflow

1. `feat(watchdog): lib/harness-heartbeat.mjs probe + 单元测试`

## Phase 4：harness self-handover 流程

STATUS: implemented 2026-04-19

**范围**：最大的 Phase，依赖 Phase 2 (ipc_spawn host) + Phase 3 (probe 触发信号)。

### 触发条件（来自 Phase 3 状态机）

- harness 10min 无响应 → `state: down`
- harness 自报 65%+ context → `state: down`

### 流程

1. watchdog 状态机 `harness → down` → 调用 `triggerHarnessSelfHandover()`
2. `triggerHarnessSelfHandover` 流程：
   a. 读 `D:/workspace/ai/research/xiheAi/xihe-tianshu-harness/.harness-checkpoint.md`
   b. 读 `D:/workspace/ai/research/xiheAi/xihe-company-brain/portfolio/STATUS.md`
   c. 按 HANDOVER schema v2（6-8 字段）生成 `HANDOVER-<timestamp>.md`
   d. 通过 Hub 自带的 bin/feishu-reply（或直接 git）把 HANDOVER + checkpoint 提交到 xihe-tianshu-harness
   e. 调 `ipc_spawn("harness", task=<读 HANDOVER>, model="opus", host="wt")` 起新 harness
   f. 新 harness 连入 → release-rebind（Phase 1）继承旧 harness 的 name + topic + 缓冲消息
   g. 新 harness 读 HANDOVER 续推

### 防递归

- `lib/lineage.mjs` 追 `parent_session_id` 链（ADR-002 §九 借鉴 Clawith trigger_daemon 的深度限制）
- 同一 session 链深度 > 3 或 1 小时内 self-handover > 2 次 → 停止递归 + IPC 告警"手动介入"

### 极端情况

ADR-002 §五 说 "jianmu-pm 自己挂 → hub-daemon 重启她 → 她重启第一件事查 harness 状态补 spawn"。
- 实现：watchdog 启动时查 `.harness-checkpoint.md` 修改时间 + 检查 harness 是否在线
- checkpoint 旧 + harness 不在线 → 立即触发 self-handover（而不是等 probe 周期）

### 测试

- 集成测试 `tests/integration/harness-self-handover.test.mjs`：
  - 模拟 harness down → mock `.harness-checkpoint.md` → 验证 HANDOVER 文件生成 + ipc_spawn 被调用（参数正确）
  - 递归限制：模拟链深度 4 → 应触发告警不 spawn
  - Phase 1 联调：spawn 后新 session 连入走 release-rebind 继承 topic

2. `feat(lineage): lib/lineage.mjs parent_session_id 链 + 深度/频次限制`
