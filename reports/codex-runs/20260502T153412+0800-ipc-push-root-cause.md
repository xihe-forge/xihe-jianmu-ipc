## Direction 1: Hub delivered semantics and ack

Finding: root cause class confirmed.

Current code already had a receiver-side ack, but the ack was sent too early:

- `lib/router.mjs` persisted routed messages with `status: delivered` immediately via `saveMessage()`.
- `mcp-server.mjs` sent `ackInboundMessage(msg)` immediately after `pushChannelNotification(msg)`.
- Before this fix, `pushChannelNotification` returned nothing. In the pre-init case it only queued the payload, so Hub could receive ack before `server.notification()` had even run.
- `inbox` flush path pushed channel notifications but did not ack individual buffered messages at all.

Fix shipped:

- `lib/channel-notification.mjs`: `pushChannelNotification()` now returns a Promise. Pre-init calls resolve only after `markInitialized()` flushes and `server.notification()` succeeds.
- `mcp-server.mjs`: realtime `message` and buffered `inbox` messages ack Hub only after notification send succeeds.
- `lib/db.mjs` + `lib/router.mjs` + `hub.mjs`: messages are persisted as `pending`, ack flips to `delivered`, ack timeout flips to `unacked` (`IPC_ACK_TIMEOUT_MS`, default 10s).
- `hub.mjs`: fixed ack notify guard so HTTP senders without a WebSocket do not trigger a spurious `ws.send` error.

Compatibility:

- Wire protocol remains compatible. Existing clients may keep sending `{type:"ack", messageId}`.
- Old/no-ack clients no longer create false `delivered`; they become `unacked` after timeout.
- `saveMessage(msg)` default remains `delivered` for direct DB callers; router opts into pending semantics.

## Direction 2: MCP initialize handshake drop

Finding: partially fixed before; this task closes the remaining ack semantic hole.

Evidence:

- `CLAUDE.md` and ADR-009 document the 2026-04-24 fix: pre-init queue in `lib/channel-notification.mjs`, flushed by `server.oninitialized`.
- Current factory already queues pre-init notifications.
- The missing piece was not the queue itself; it was that callers had no completion signal, so mcp-server acked when queued, not when flushed/sent.

Live dogfood caught the exact cold-window path:

- Session: `codex_live_model_153412`
- Message: `msg_1777708224286_94aa53`
- Trace: `channel_push_begin` with `mcp_initialized:false`
- Trace: `mcp_initialized_flush_begin queued_count:1`
- Trace: `channel_notification_send_ok`
- Trace: `ack_sent`
- Hub DB: message status `delivered`

## Direction 4: spawn path difference

Finding: no direct code-level divergence explaining push loss; remaining difference is launch environment/parent UI.

Evidence:

- `bin/install.ps1 function ipc` launches `claude.exe --dangerously-skip-permissions --dangerously-load-development-channels server:ipc` through `bin/claude-stdin-auto-accept.mjs`, with `IPC_NAME` in env and cwd `D:\workspace\ai\research\xiheAi`.
- `ipc_spawn` Claude `host=wt` path builds the same Claude channel args and also uses `IPC_NAME` env plus `bin/claude-stdin-auto-accept.mjs`; it adds task prompt injection and trust patching.
- `mcp-wrapper.mjs` lifecycle leak was fixed in `2577539`; wrapper now restarts child after unintentional exit and removes stale handlers.

Conclusion:

- The prior pc-pet vs taiwei split is consistent with timing/handshake exposure rather than different wire protocol.
- This fix makes both paths use the same ack truth: no delivered until receiver mcp-server has actually pushed the notification out of its pre-init queue.
